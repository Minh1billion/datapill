Getting Started with the datapill CLI

Last updated: May 2, 2026  |  Version: 0.1.0

datapill is a command-line tool for ingesting, inspecting, and managing data pipelines. It connects to databases, object storage, message queues, and REST APIs - streaming data into tracked artifacts you can query, compare, and version.

ℹ️ What you'll learn
By the end of this guide you will be able to install datapill, connect to a data source, run your first ingestion, and inspect the resulting artifact from the command line.

Core concepts

  • Source - A named connector type (postgres, s3, kafka, etc.)
  • Config file - A .json file holding connection parameters for a source.
  • Artifact - A versioned record of a pipeline run, optionally materialized as Parquet on disk.
  • Store - A local .datapill/ directory that tracks all artifacts via DuckDB.

Installation

datapill requires Python 3.11 or newer.

Install from PyPI

bash
$ pip install datapill

Verify the installation

bash
$ dp --version
datapill 0.1.0

$ dp --help
Usage: dp [OPTIONS] COMMAND [ARGS]...

  datapill - lightweight data ingestion and pipeline toolkit.

Commands:
  artifact  inspect and manage pipeline artifacts
  ingest    ingest data from a source into datapill

Quick Start

The following steps walk you through ingesting a table from PostgreSQL end-to-end.

  1. Create a config file Save your connection parameters as a .json file. Keep this file out of version control.
    postgres.json
    {
  "host":     "db.example.com",
  "port":      5432,
  "database": "sales_db",
  "user":     "analyst",
  "password": "s3cr3t"
}
  2. Check the connection Verify datapill can reach your database before running a full ingest.
    bash
    $ dp ingest check postgres --config postgres.json
  ✔ connected  latency 4ms
  3. Ingest a table Run a full ingest and materialize the result to a Parquet artifact.
    bash
    $ dp ingest run postgres \
    --config      postgres.json \
    --table       orders \
    --materialize \
    --schema

  run_id   a1b2c3d4
  ref      orders:a1b2c3d4

  column          type
  ─────────────── ─────────
  id              Int64
  customer_id     Int64
  total           Float64
  created_at      Datetime

  artifact  .datapill/artifacts/orders/a1b2c3d4.parquet
  4. Inspect the artifact View metadata for the run you just created.
    bash
    $ dp artifact show a1b2c3d4

  run          a1b2c3d4
  pipeline     orders
  timestamp    2026-05-02 09:14
  sample       no
  materialized yes
  path         artifacts/orders/a1b2c3d4.parquet

datapill ingest

The ingest command group handles running and validating data ingestion pipelines.

ingest run

Read data from a source and optionally store it as a versioned artifact.

synopsis
dp ingest run SOURCE --config FILE [OPTIONS]

Options

Argument / FlagTypeDefaultDescription
SOURCE requiredstring…source type: postgres, mysql, sqlite, s3, local, kafka, rest
--config / -c optionalstring-path to config .json file (required for postgres, mysql, sqlite, kafka, rest, s3; optional for local)
--table / -t optionalstring-table name (postgres, mysql, sqlite)
--query / -q optionalstring-raw SQL query (postgres, mysql, sqlite)
--topic optionalstring-kafka topic
--path / -p optionalstring-file path (local: full path to file; s3: key within bucket)
--endpoint / -e optionalstring-REST endpoint path
--params optionalstring-path to params .json file
--sample optionalflagfalseread a sample instead of full data
--sample-size optionalint10000number of rows to sample
--batch-size optionalint-rows per streaming batch
--materialize / -m optionalflagfalsewrite output to parquet artifact
--store optionalstring.datapillartifact store directory
--schema optionalflagfalseprint column schema after ingest
--mkdir optionalflagfalsecreate base_path directory if it does not exist (local source only)

Examples

bash
$ dp ingest run postgres \
    --config  pg.json \
    --query   "SELECT id, email FROM users WHERE active = true" \
    --materialize \
    --schema
bash
$ dp ingest run mysql \
    --config mysql.json \
    --table  products
bash
$ dp ingest run postgres \
    --config      pg.json \
    --table       events \
    --sample \
    --sample-size 5000 \
    --schema
# Useful for validating schema before reading the full table
bash
$ dp ingest run s3 \
    --config s3.json \
    --path   "data/2026/orders.parquet" \
    --materialize

ingest check

Test a connector's credentials and network reachability without ingesting any data. Reports round-trip latency on success.

synopsis
dp ingest check SOURCE --config FILE
bash
$ dp ingest check kafka --config kafka.json
  ✔ connected  latency 12ms

$ dp ingest check postgres --config bad.json
  ✖ connection failed: password authentication failed for user "analyst"

Options

Argument / FlagTypeDefaultDescription
SOURCE requiredstring…source type
--config / -c optionalstring-path to config .json file
--path / -p optionalstring-file path (for local source)
--mkdir optionalflagfalsecreate base_path directory if it does not exist (local source only)

datapill artifact

Manage the artifact store - list, inspect, trace lineage, and clean up pipeline runs.

artifact list

synopsis
dp artifact list [--pipeline NAME] [--limit N] [--store PATH]
bash
$ dp artifact list --pipeline orders --limit 5

 run        pipeline  when              sample  mat
 ────────── ───────── ────────────────  ──────  ───
 a1b2c3d4   orders    2026-05-02 09:14  ·       y
 9f8e7d6c   orders    2026-05-01 22:01  ·       ·
 3a4b5c6d   orders    2026-04-30 18:40  y       ·

artifact show

Display full metadata for a single artifact by run ID.

synopsis
dp artifact show RUN_ID [--store PATH]

artifact lineage

Trace the ancestry chain of an artifact back to its root. Useful when pipelines are chained.

bash
$ dp artifact lineage a1b2c3d4

raw_ingest   0000ffff  2026-04-29 08:00
  └─ clean   1234abcd  2026-04-29 08:15
    └─ orders   a1b2c3d4  2026-05-02 09:14

artifact delete

Delete a single artifact and its associated files on disk. Prompts for confirmation unless --yes is passed.

bash
$ dp artifact delete a1b2c3d4 --yes
  ✔ deleted  a1b2c3d4

artifact purge

Bulk-delete artifacts, keeping the N most recent per pipeline. Use --samples-only to target only sample runs.

bash
$ dp artifact purge --pipeline orders --keep 3 --yes
  ✔ purged  7 artifact(s)

artifact usage

Show total artifact count and disk usage for the store.

bash
$ dp artifact usage

  artifacts    42
  disk usage   1.3 GB

datapill profile

Profile a dataset artifact — computes column statistics, histograms, correlations, and data quality warnings.

profile run

Run a full statistical profile on a parent ingest or preprocess artifact.

synopsis
dp profile run PARENT_RUN_ID [OPTIONS]

Options

Argument / FlagTypeDefaultDescription
PARENT_RUN_ID requiredstring…run id of the parent artifact (ingest or preprocess)
--mode optionalstringfull'full' computes histograms/correlations/patterns, 'summary' is faster
--sample-strategy optionalstring-sampling strategy: none | random | reservoir
--sample-size optionalint100000number of rows to use when sample-strategy != none
--chunk-size optionalint100000rows per processing chunk
--correlation optionalstringpearsoncorrelation method: pearson | spearman | none
--correlation-threshold optionalfloat0.3only show pairs with |r| >= threshold
--no-patterns optionalflagfalseskip string pattern detection
--histogram-bins optionalint20number of histogram bins for numeric columns
--cardinality-limit optionalint100max distinct values to show in top_values
--store optionalstring.datapillartifact store directory
--schema optionalflagfalseprint column schema after profiling

profile show

Display a saved profile result by run ID.

synopsis
dp profile show RUN_ID [--store PATH]

Omit access_key and secret_key to use the default credential chain (environment variables, ~/.aws/credentials, or an EC2 instance profile).

Streaming Large Datasets

By default, datapill loads an entire result set into memory. For large tables, add --batch-size to enable chunked streaming:

bash
$ dp ingest run postgres \
    --config     pg.json \
    --table      events \
    --batch-size 100000 \
    --materialize
⚠️ Memory note
Even with streaming, each batch is held in memory until written. Set --batch-size low enough to fit comfortably within your available RAM.

Recommended batch sizes by source:

SourceRecommended batch-sizeNotes
PostgreSQL100 000 – 500 000Server-side cursor via asyncpg
MySQL50 000 – 200 000SSCursor for row-by-row fetch
SQLite50 000DuckDB Arrow reader handles chunking
S3 (CSV / JSON)8 MB chunksControlled by chunk_size in config
Kafka500 messagesControlled by max_poll_records in config

Working with Artifacts

Every ingest run creates an artifact entry in .datapill/store.db. With --materialize, the data is also written to .datapill/artifacts/<pipeline>/<run_id>.parquet.

Reading a materialized artifact with DuckDB

python
import duckdb

path = ".datapill/artifacts/orders/a1b2c3d4.parquet"
df   = duckdb.sql(f"SELECT * FROM read_parquet('{path}') LIMIT 10").df()
print(df)

Recommended workflow

  • Use --sample first to validate schema and data quality cheaply.
  • Run a full ingest with --materialize once the sample looks correct.
  • Use artifact lineage to trace how a derived dataset was produced.
  • Schedule artifact purge --keep 5 in CI to prevent unbounded disk growth.
