DataPrep CLI - dp

A command-line tool for ingesting, profiling, preprocessing, and exporting data across PostgreSQL, MySQL, S3, Kafka, REST APIs, and local files.

Python ≥ 3.11 Polars · PyArrow asyncio-native entry: dp

Installation

Install the package in editable mode from the project root. The dp entry point is registered automatically via pyproject.toml.

bash
# Install with dev dependencies
pip install -e ".[dev]"

# Verify
dp --help

Quick Start

The minimal path from a CSV file to a profiled, preprocessed, exported dataset.

  1. Ingest a local CSV file into the artifact store.
  2. Profile the ingested dataset to understand its structure.
  3. Apply preprocessing transformations.
  4. Export the result to Parquet or write back to a database.
bash
dp ingest --source local_file --path data.csv
# -> Run ID:  abc123
# -> Artifact: abc123_ingest_output

dp profile --input abc123

dp preprocess --input abc123 --pipeline pipeline.json
# -> Run ID:  def456
# -> Output:  def456_preprocess_output

dp export --input def456 --format parquet --out-path output/result.parquet

Artifact System

Every pipeline command saves its outputs as artifacts - files stored in the artifact store directory (default: src/dataprep/artifacts). Understanding artifacts is essential for chaining commands together.

Run ID vs Artifact ID

Each pipeline run generates a short run ID (e.g. f1662de6). The run produces one or more artifacts, each named <run_id>_<suffix>.

Example - preprocess run
Run ID f1662de6
Output f1662de6_preprocess_output <- data
Config f1662de6_preprocess_config <- step config snapshot

Passing input to commands

Any command that takes --input accepts either the run ID or a full artifact ID. When you pass a run ID, the system picks the most relevant artifact automatically based on the target command.

What you passHow it resolves
run_id only f1662de6Selects the best artifact for the next command - e.g. profile prefers ingest_output, export prefers preprocess_output
full artifact ID f1662de6_ingest_outputUses that exact artifact, no resolution needed
ℹ Use dp list to see all artifacts in the store along with their run IDs, features, and creation timestamps.

Artifact types by feature

FeatureArtifacts producedFormat
ingest<run_id>_ingest_output, <run_id>_ingest_schemaParquet, JSON
profile<profile_id>_detail, <profile_id>_summaryJSON
preprocess<run_id>_preprocess_output, <run_id>_preprocess_configParquet, JSON

Kafka - Produce test messages

bash
# Create topic first (use // prefix on Git Bash to prevent path conversion)
docker compose -f docker-compose.test.yml exec -T kafka \
  //opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --create --topic test-topic \
  --partitions 1 --replication-factor 1

# Produce 100 JSON messages
python - << 'EOF'
import asyncio, json
from aiokafka import AIOKafkaProducer

async def produce():
    p = AIOKafkaProducer(bootstrap_servers="localhost:9092")
    await p.start()
    for i in range(100):
        msg = json.dumps({"id": i, "product_name": f"Product_{i}",
            "category": ["Electronics","Clothing","Food","Books"][i%4],
            "price": round(i*10.5,2), "quantity": i%50}).encode()
        await p.send("test-topic", msg)
    await p.stop()
    print("Produced 100 messages")

asyncio.run(produce())
EOF

dp connector

Interact directly with a data source without going through a full pipeline. Useful for verifying connectivity, inspecting schema, and performing one-off operations like uploads, truncates, or Kafka message production before committing to a full ingest run.

ActionSupported SourcesDescription
testallVerify connectivity and measure latency
schemaallInspect column names, types, and row estimate
uploads3, local_fileUpload a local file to S3 or copy to a local destination
downloads3Download an S3 object to a local path
lists3List objects in a bucket with optional prefix filter
execpostgresql, mysqlExecute a raw SQL statement (DELETE, UPDATE, etc.)
truncatepostgresql, mysqlTruncate a table
producekafkaSend rows from a JSON/CSV file to a Kafka topic

test

Checks that the connector can reach the target system and reports round-trip latency. Run this first to rule out connectivity issues before ingesting.

bash
dp connector test --source postgresql --config tests/fixtures/pg_config.json
dp connector test --source mysql       --config tests/fixtures/mysql_config.json
dp connector test --source s3          --config tests/fixtures/s3_config.json
dp connector test --source kafka       --config tests/fixtures/kafka_config.json
dp connector test --source rest        --config tests/fixtures/rest_config.json

schema

Returns column names, data types, nullability, and an estimated row count. Useful for checking that a table or file has the expected shape before ingesting.

bash
dp connector schema --source postgresql --config tests/fixtures/pg_config.json --table orders
dp connector schema --source mysql       --config tests/fixtures/mysql_config.json --table products
dp connector schema --source s3          --config tests/fixtures/s3_config.json --url s3://testbucket/input/data.csv
dp connector schema --source local_file  --path tests/fixtures/data.csv

upload

Reads a local file (CSV, Parquet, JSON, JSONL, or Excel) and writes it to an S3 destination or another local path. The source format is inferred from the file extension.

bash
dp connector upload \
  --source   s3 \
  --config   tests/fixtures/s3_config.json \
  --src-path tests/fixtures/data.csv \
  --dest-url s3://testbucket/upload/data.csv

download

Downloads an S3 object to a local file. The destination format is inferred from the --out-path extension. Intermediate directories are created automatically.

bash
dp connector download \
  --source   s3 \
  --config   tests/fixtures/s3_config.json \
  --url      s3://testbucket/input/data.csv \
  --out-path output/downloaded.csv

list

Lists all objects in an S3 bucket, showing key, size, and last modified time. Use --prefix to narrow down to a specific folder or key prefix.

bash
# List all objects
dp connector list --source s3 --config tests/fixtures/s3_config.json

# Filter by prefix
dp connector list --source s3 --config tests/fixtures/s3_config.json --prefix input/

exec

Executes a raw SQL statement against a PostgreSQL or MySQL database and commits the transaction. Intended for one-off DML operations such as deletes or updates - not for SELECT queries.

bash
# PostgreSQL - delete rows
dp connector exec \
  --source postgresql \
  --config tests/fixtures/pg_config.json \
  --sql "DELETE FROM orders WHERE status='cancelled'"

# MySQL - update rows
dp connector exec \
  --source mysql \
  --config tests/fixtures/mysql_config.json \
  --sql "UPDATE products SET stock=0 WHERE price > 900"

truncate

Issues a TRUNCATE TABLE against the specified table. Faster than a full delete and resets auto-increment counters on MySQL. Use before a replace-mode export to avoid constraint issues.

bash
dp connector truncate --source postgresql --config tests/fixtures/pg_config.json --table orders_processed
dp connector truncate --source mysql       --config tests/fixtures/mysql_config.json --table products_processed

produce

Reads a local JSON or CSV file and publishes each row as a JSON-encoded message to a Kafka topic. Optionally uses a column value as the message key for partition routing.

bash
dp connector produce \
  --source     kafka \
  --config     tests/fixtures/kafka_config.json \
  --topic      test-topic \
  --file       tests/fixtures/data.csv

# With a message key column
dp connector produce \
  --source     kafka \
  --config     tests/fixtures/kafka_config.json \
  --topic      test-topic \
  --file       tests/fixtures/data.csv \
  --key-column id

dp ingest

Reads data from a connector in streaming batches and persists it as a Parquet artifact in the artifact store. Also saves a schema snapshot (column names, types, row count) as a companion JSON artifact. The resulting artifact ID is what you pass to profile, preprocess, or export.

For sources with inconsistent column schemas across batches (e.g. paginated REST APIs), ingest coerces every chunk to the schema of the first chunk so the final Parquet file is consistent.

OptionDefaultDescription
--source, -srequiredConnector type: local_file | postgresql | mysql | s3 | rest | kafka
--config, -c-Path to connector JSON config file
--path-Local file path (local_file)
--table-Table name (postgresql | mysql)
--url-S3 URL, e.g. s3://bucket/key.parquet
--topic-Kafka topic name
--endpoint-REST endpoint path, e.g. posts
--out, -osrc/dataprep/artifactsArtifact store directory
--limit, -n-Max rows to read
--batch-size50 000Rows per streaming batch
--max-records-Max messages consumed (kafka)
bash
# Local CSV
dp ingest --source local_file --path tests/fixtures/data.csv

# Local CSV - large file with row limit
dp ingest --source local_file --path tests/fixtures/data_1M.csv \
  --batch-size 100000 --limit 50000

# PostgreSQL
dp ingest --source postgresql --config tests/fixtures/pg_config.json --table orders

# MySQL
dp ingest --source mysql --config tests/fixtures/mysql_config.json --table products

# S3
dp ingest --source s3 --config tests/fixtures/s3_config.json \
  --url s3://testbucket/input/data.csv

# REST API (no leading slash on Git Bash)
dp ingest --source rest --config tests/fixtures/rest_config.json --endpoint posts

# Kafka - consume up to 100 messages
dp ingest --source kafka --config tests/fixtures/kafka_config.json \
  --topic test-topic --max-records 100

dp profile

Computes descriptive statistics for every column in the dataset and saves two artifacts: a detailed JSON (per-column histograms, percentiles, top values, pattern matches) and a lighter summary JSON (null rates, distinct counts, min/max, warnings). A column-summary table is printed to the terminal at the end of the run.

For large datasets, use --sample-strategy random or reservoir to profile a representative subset instead of loading the full file into memory.

OptionDefaultDescription
--input, -irequiredRun ID or full artifact ID from a previous ingest or preprocess
--mode, -mfullfull - all stats; summary - lightweight pass
--sample-strategynonenone | random | reservoir
--sample-size100 000Rows to sample when strategy ≠ none
--correlationpearsonCorrelation method: pearson | spearman | none
--out, -osrc/dataprep/artifactsArtifact store directory
bash
# Full profile using run ID
dp profile --input 72e0548a

# Or using full artifact ID
dp profile --input 72e0548a_ingest_output

# Full profile with random sampling (large datasets)
dp profile --input 72e0548a \
  --mode full \
  --sample-strategy random \
  --sample-size 10000 \
  --correlation pearson

# Profile directly from a parquet file
dp profile --input output/result.parquet --mode summary

dp preprocess

Applies a sequence of transformation steps to an ingested dataset, defined in a JSON pipeline file. Steps run in order; each step receives the output of the previous one. The pipeline also detects common conflicts (e.g. imputing a column that will later be dropped, or applying two clipping strategies to the same column) and surfaces them as warnings after the run.

Use --dry-run to validate your pipeline on the first 1 000 rows before committing. Use --checkpoint to save intermediate Parquet files after each step, which is useful for debugging a long pipeline.

OptionDefaultDescription
--input, -irequiredRun ID or full artifact ID from a previous ingest or preprocess
--pipeline, -prequiredPath to pipeline JSON config file
--dry-runfalseRun on first 1 000 rows only - no artifact is saved
--checkpointfalseSave a Parquet file after each step for inspection
--out, -osrc/dataprep/artifactsArtifact store directory

Pipeline JSON format

json - tests/fixtures/preprocess_pipeline.json
{
  "steps": [
    {"type": "impute_mean",    "scope": {"columns": ["price", "quantity"]}},
    {"type": "clip_iqr",       "scope": {"columns": ["price"]}},
    {"type": "standard_scaler", "scope": {"columns": ["price", "quantity"]}}
  ]
}
bash
# Dry run first to validate the pipeline
dp preprocess \
  --input    72e0548a \
  --pipeline tests/fixtures/preprocess_pipeline.json \
  --dry-run

# Full run using run ID
dp preprocess \
  --input    72e0548a \
  --pipeline tests/fixtures/preprocess_pipeline.json

# Full run using explicit artifact ID, with checkpoints
dp preprocess \
  --input    72e0548a_ingest_output \
  --pipeline tests/fixtures/preprocess_pipeline.json \
  --checkpoint

After a successful run, the terminal shows:

output
  Output:  f1662de6_preprocess_output
  Config:  f1662de6_preprocess_config
  Run ID:  f1662de6

Pass either the Run ID (f1662de6) or the full Output artifact ID to the next command.

dp export

Writes a dataset artifact to a local file or pushes it back to a database or S3. Supports three write modes - replace, append, and upsert - to control how existing data in the target is handled.

For database write-backs, rows are inserted in batches (default 5 000 rows per batch). Use --dry-run to preview the first 10 rows without writing anything.

OptionDefaultDescription
--input, -irequiredRun ID or full artifact ID from a previous ingest or preprocess
--format, -frequiredcsv | parquet | json | jsonl | excel
--out-path-Destination file path (required when not using --connector)
--connector, -c-Write-back connector config JSON with a source field
--write-modereplacereplace | append | upsert
--primary-keys-Comma-separated key columns for upsert
--compression-snappy | zstd | gzip (parquet only)
--dry-runfalsePrint first 10 rows, skip write
--out, -osrc/dataprep/artifactsArtifact store directory
bash
# Dry run - preview 10 rows using run ID
dp export --input f1662de6 --format parquet \
  --out-path output/result.parquet --dry-run

# Export to Parquet
dp export --input f1662de6 --format parquet \
  --out-path output/result.parquet

# Export to CSV
dp export --input f1662de6 --format csv \
  --out-path output/result.csv

# Parquet with zstd compression
dp export --input f1662de6_preprocess_output --format parquet \
  --out-path output/result.zstd.parquet --compression zstd

# Write back to PostgreSQL - replace
dp export --input f1662de6 --format parquet \
  --connector tests/fixtures/pg_write_config.json --write-mode replace

# Write back to PostgreSQL - upsert
dp export --input f1662de6 --format parquet \
  --connector tests/fixtures/pg_write_config.json \
  --write-mode upsert --primary-keys id

# Write back to MySQL - replace
dp export --input d7859e9e --format parquet \
  --connector tests/fixtures/mysql_write_config.json --write-mode replace

# Write back to S3
dp export --input f1662de6 --format parquet \
  --connector tests/fixtures/s3_write_config.json
⚠ append + primary key constraint: Using --write-mode append on a table with a primary key will fail if duplicate IDs exist. Use upsert with --primary-keys instead, or truncate the table first.

dp list

Lists artifacts stored in the artifact store, sorted by creation time (newest first). Each row shows the full artifact ID, the run ID it belongs to, which feature produced it, the file extension, and the creation timestamp. Use this to find run IDs or artifact IDs to pass to other commands.

OptionDefaultDescription
--out, -osrc/dataprep/artifactsArtifact store directory to inspect
--feature, -f-Filter by feature: ingest | profile | preprocess
--limit, -n20Max number of artifacts to display
bash
# List the 20 most recent artifacts
dp list

# Show only ingest artifacts
dp list --feature ingest

# Show only preprocess artifacts, up to 50
dp list --feature preprocess --limit 50

# Inspect a different artifact store
dp list --out /tmp/dp_artifacts

dp run

Runs an ingest followed immediately by a profile in a single command, driven by a JSON config file. Useful for automating a repeatable ingestion + quality-check workflow without manually chaining two commands and copying artifact IDs.

Config file format

json - tests/fixtures/run_config_pg.json
{
  "source": "postgresql",
  "connector": {
    "host": "localhost", "port": 5433,
    "database": "testdb", "user": "testuser", "password": "testpass"
  },
  "query":   { "table": "orders" },
  "ingest":  { "batch_size": 10000 },
  "profile": {
    "mode": "full", "sample_strategy": "random",
    "sample_size": 10000, "correlation": "pearson"
  }
}
bash
dp run tests/fixtures/run_config_pg.json

# Custom artifact output directory
dp run tests/fixtures/run_config_pg.json --out /tmp/dp_artifacts

Walkthrough - PostgreSQL end-to-end

bash
# 1. Verify connection
dp connector test   --source postgresql --config tests/fixtures/pg_config.json

# 3. Profile - use run ID directly

  --sample-strategy random --sample-size 10000

  --checkpoint

# 5a. Export to Parquet

  --out-path output/result.parquet

# 5b. Or write back to PostgreSQL

  --connector tests/fixtures/pg_write_config.json \
  --write-mode upsert --primary-keys id

Write Modes

All three write modes require the target table to already exist in the database. The export pipeline does not create tables automatically. Create the table with a schema that matches the dataset you are exporting before running dp export.

Setup - create the target table first

The column types must be compatible with the exported data. If you preprocessed with standard_scaler or similar steps, numeric columns will have decimal values - use NUMERIC(10,6) rather than INTEGER.

bash - PostgreSQL
docker compose -f docker-compose.test.yml exec -T postgres \
  psql -U testuser -d testdb \
  -c "
CREATE TABLE IF NOT EXISTS orders_processed (
    id INTEGER PRIMARY KEY,
    product_name VARCHAR(100),
    category VARCHAR(50),
    price NUMERIC(10,6),
    quantity NUMERIC(10,6),
    status VARCHAR(20),
    created_at TIMESTAMP
);
"
bash - MySQL
docker compose -f docker-compose.test.yml exec -T mysql \
  mysql -u testuser -ptestpass testdb \
  -e "
CREATE TABLE IF NOT EXISTS orders_processed (
    id INT PRIMARY KEY,
    product_name VARCHAR(100),
    category VARCHAR(50),
    price DECIMAL(10,6),
    quantity DECIMAL(10,6),
    status VARCHAR(20),
    created_at DATETIME
);
"

Behaviour by mode

ModeBehaviourRequires
replaceTRUNCATE the target table, then INSERT all rows. Re-running produces the same row count.Table must exist.
appendINSERT all rows without touching existing data. Fails immediately if any row violates a unique constraint.Table must exist. Use only when the source data contains no IDs already present in the target.
upsertINSERT … ON CONFLICT DO UPDATE for each row. Existing rows are updated, new rows are inserted. Re-running is safe and idempotent.Table must have a primary key or unique constraint on the --primary-keys columns.

Testing each mode

bash
# replace - run twice, count stays the same
dp export --input <run_id> --format parquet \
  --connector tests/fixtures/pg_write_config.json --write-mode replace

dp export --input <run_id> --format parquet \
  --connector tests/fixtures/pg_write_config.json --write-mode replace

# append - will ERROR because IDs already exist after replace above
dp export --input <run_id> --format parquet \
  --connector tests/fixtures/pg_write_config.json --write-mode append

# upsert - safe to re-run, count stays the same
dp export --input <run_id> --format parquet \
  --connector tests/fixtures/pg_write_config.json \
  --write-mode upsert --primary-keys id
ℹ Verify row count after each mode with: psql -U testuser -d testdb -c "SELECT COUNT(*) FROM orders_processed;"
✓When in doubt, use upsert with a primary key. It is idempotent and safe to re-run.

Pipeline Steps

All numeric steps enforce that the target columns are numeric - a TypeError is raised at runtime otherwise. Steps that are not numeric-only (e.g. drop_nulls, drop_duplicates) can be applied to any column type.

Step typeDescriptionColumn types
impute_meanReplace nulls with the column mean.Numeric
impute_medianReplace nulls with the column median.Numeric
impute_modeReplace nulls with the most frequent value.Any
clip_iqrClip outliers to [Q1 − 1.5×IQR, Q3 + 1.5×IQR].Numeric
standard_scalerStandardise to zero mean and unit variance.Numeric
min_max_scalerScale values to the [0, 1] range.Numeric
log_transformApply log1p transformation. Values must be non-negative.Non-negative numeric
drop_duplicatesRemove exact duplicate rows.All (or subset)
drop_nullsDrop rows where any of the specified columns are null.Any
⚠ The pipeline detects certain step conflicts and surfaces them as warnings after the run - for example, imputing a column that is later dropped by drop_nulls, or applying both clip_iqr and clip_zscore to the same column. Warnings do not stop the run; review them before re-running on production data.
